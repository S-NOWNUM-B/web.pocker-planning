# Вопросы и ответы по проекту Pocker Planning

### 1. Какие технологии используются в моём проекте? Для чего они используются?

Проект представляет собой современный fullstack-стек, организованный в виде монорепозитория.

#### Frontend (Клиентская часть)
- **React 19**: Основная библиотека для создания пользовательского интерфейса.
- **TypeScript**: Обеспечивает строгую типизацию, что снижает количество ошибок и упрощает рефакторинг.
- **Vite 6**: Инструмент сборки, обеспечивающий очень быструю горячую перезагрузку (HMR) и оптимизированную сборку.
- **Tailwind CSS**: CSS-фреймворк для быстрой и гибкой стилизации интерфейса с помощью утилитных классов.
- **TanStack Query (React Query)**: Управление серверным состоянием, кэширование данных и синхронизация с API.
- **Zustand**: Легковесный менеджер глобального клиентского состояния (например, для хранения данных сессии).
- **React Router**: Маршрутизация приложения (используется в Framework Mode).
- **Zod**: Схема-валидация данных (используется вместе с React Hook Form для валидации форм).

#### Backend (Серверная часть)
- **Python 3.13**: Язык программирования, обеспечивающий высокую скорость разработки и богатую экосистему.
- **FastAPI**: Высокопроизводительный фреймворк для создания REST API и работы с WebSockets в реальном времени.
- **PostgreSQL 16**: Надежная реляционная база данных для постоянного хранения пользователей, комнат и истории голосований.
- **SQLAlchemy 2.0**: Современная ORM для взаимодействия с базой данных с использованием Python-объектов.
- **Alembic**: Инструмент для управления миграциями базы данных (изменение схемы БД без потери данных).
- **Redis**: Используется для быстрого кэширования и управления сессиями.

#### Инфраструктура и Инструменты
- **Turbo (Turborepo)**: Система управления монорепозиторием, которая ускоряет сборку и тестирование за счет кэширования.
- **pnpm**: Быстрый и эффективный менеджер пакетов.
- **Docker & Docker Compose**: Контейнеризация всех сервисов для обеспечения идентичности окружения при разработке и развертывании.
- **Nginx**: Веб-сервер для раздачи статики фронтенда и проксирования запросов.

### 2. Для чего нужны папки apps, infrastructure, packages? Как с ними правильно взаимодействовать, работать и применять?

Эта структура характерна для **монорепозитория**. Она позволяет держать весь код проекта (фронтенд, бэкенд, общие типы и настройки инфраструктуры) в одном месте, обеспечивая при этом четкое разделение ответственности.

#### apps/ (Приложения)
Здесь живут полноценные, запускаемые приложения.
- **Что внутри:** frontend (React-приложение) и backend (FastAPI-сервер).
- **Как работать:** 
    - Для запуска всего стека используйте корневые команды pnpm dev.
    - Если нужно работать только с одной частью, переходите в соответствующую папку (например, cd apps/backend для работы с Python-окружением и миграциями Alembic).
    - Каждое приложение в этой папке является независимым сервисом с собственным package.json или requirements.txt.

#### packages/ (Общие пакеты/библиотеки)
Здесь хранится код, который может использоваться несколькими приложениями из папки apps.
- **Что внутри:** Например, packages/shared содержит общие TypeScript-типы, константы или утилиты, которые нужны и фронтенду, и другим возможным TS-сервисам.
- **Как работать:** 
    - Если вам нужно изменить структуру данных (например, добавить поле в объект User), которое используется в разных частях системы, делайте это здесь.
    - Приложения в apps/ импортируют код из packages/ как обычные npm-пакеты (благодаря pnpm workspaces).

#### infrastructure/ (Инфраструктура)
Здесь хранятся конфигурации для развертывания, оркестрации и настройки среды.
- **Что внутри:** Конфигурации Docker, Docker Compose, настройки CI/CD, скрипты деплоя.
- **Как работать:** 
    - Используйте файлы из этой папки для локального запуска всей системы одной командой (docker compose up).
    - Здесь определяются связи между сервисами (например, как бэкенд соединяется с базой данных PostgreSQL и Redis).

### Итоговая схема взаимодействия:
packages/shared -> (импортируется в) -> apps/frontend & apps/backend -> (оборачиваются в контейнеры через) -> infrastructure/

### 3. Что такое FSD архитектура? Как с ней правильно работать и что за что внутри неё отвечает?

**FSD (Feature-Sliced Design)** — это архитектурная методология для фронтенд-приложений, которая разделяет код не по типу файлов (например, все компоненты в одной папке, все хуки в другой), а по **бизнес-ценности** и **уровням ответственности**.

Основной принцип: **строгая иерархия**. Слой, который находится выше, может импортировать код из слоев ниже, но никогда наоборот.

#### Слоеная структура (от самого верхнего к самому нижнему):

1. **app/** (Приложение)
   - **Ответственность:** Глобальная инициализация приложения.
   - **Что внутри:** Провайдеры (QueryProvider, SessionProvider), глобальные стили, конфигурация роутинга (routes.tsx).
   - **В вашем проекте:** apps/frontend/src/app.

2. **pages/** (Страницы)
   - **Ответственность:** Композиция страниц. Страница собирает воедино виджеты и фичи, чтобы создать полноценный экран.
   - **Что внутри:** Контент конкретных страниц (RoomPage, DashboardPage). Здесь обычно происходит привязка параметров URL к данным.
   - **В вашем проекте:** apps/frontend/src/pages.

3. **widgets/** (Виджеты)
   - **Ответственность:** Крупные, самостоятельные блоки интерфейса, объединяющие несколько фич или сущностей.
   - **Что внутри:** TaskSidebar (собирает управление задачами), RoomHeader (информация о комнате + статус).
   - **В вашем проекте:** apps/frontend/src/widgets.

4. **features/** (Фичи)
   - **Ответственность:** Конкретные пользовательские действия, несущие бизнес-ценность. «Что пользователь может сделать?».
   - **Что внутри:** voting (процесс голосования), auth (логин/регистрация), join-room (вход в комнату).
   - **В вашем проекте:** apps/frontend/src/features.

5. **entities/** (Сущности)
   - **Ответственность:** Бизнес-сущности приложения. «О чем наше приложение?». Описывают структуру данных и базовые UI-элементы для их отображения.
   - **Что внутри:** user (данные пользователя, карточка профиля), room (состояние комнаты, API запросы к комнате).
   - **В вашем проекте:** apps/frontend/src/entities.

6. **shared/** (Общий слой)
   - **Ответственность:** Переиспользуемый код, не привязанный к бизнес-логике. Инструменты, UI-kit, API-клиенты.
   - **Что внутри:** ui/Button, ui/Input, api/client.ts, общие хуки (useLocalStorage).
   - **В вашем проекте:** apps/frontend/src/shared.

#### Как с этим правильно работать:

- **Правило импортов:** Если вы находитесь в entities/user, вы можете импортировать что-то из shared. Но вы не имеете права импортировать что-то из features/auth или pages/LoginPage. Это предотвращает запутанные зависимости (циклические ссылки).
- **Где создавать новый код?**
    - Нужен новый UI-компонент (кнопка, инпут) -> shared/ui.
    - Нужно добавить новое поле в профиль пользователя -> entities/user.
    - Нужно создать функционал «удаления задачи» -> features/task-management.
    - Нужно собрать экран с результатами голосования -> pages/RoomPage -> widgets/RoomResults.

### 4. Что такое хук в React. Как он работает. Что значит кастомный хук и не кастомный хук?

**Хук (Hook)** — это специальная функция в React, которая позволяет «подцепиться» к внутренним механизмам React (состоянию, жизненному циклу, контексту) внутри функциональных компонентов. До появления хуков такие возможности были доступны только в классовых компонентах.

#### Как это работает
React отслеживает порядок вызова хуков в каждом компоненте. Когда компонент перерисовывается (re-render), React сопоставляет вызовы хуков по их порядковому номеру, чтобы вернуть правильное состояние для каждого из них. Именно поэтому есть важное правило: **хуки нельзя вызывать внутри условий, циклов или вложенных функций** — они должны всегда вызываться в одном и том же порядке при каждом рендере.

#### Не кастомные (встроенные) хуки
Это стандартные функции, которые предоставляет сама библиотека React. Они решают базовые задачи управления состоянием и побочными эффектами.
- **useState**: Позволяет компоненту «запоминать» данные между рендерами (локальное состояние).
- **useEffect**: Позволяет выполнять побочные эффекты (запросы к API, подписки, работа с DOM) после рендера.
- **useContext**: Позволяет получать данные из общего контекста без необходимости передавать пропсы через все уровни дерева компонентов.
- **useMemo / useCallback**: Оптимизация, позволяющая кэшировать вычисления или функции, чтобы избежать лишних перерендеров.
- **useRef**: Создает ссылку на DOM-элемент или хранит значение, которое не должно вызывать перерендер при изменении.

#### Кастомные хуки
**Кастомный хук** — это обычная JavaScript-функция, имя которой обязательно начинается с приставки `use` (например, `useWebSocket`, `useAuth`), и которая внутри себя вызывает другие хуки (встроенные или другие кастомные).

**Зачем они нужны?**
Их главная цель — **извлечение и переиспользование логики**. Если вам нужно использовать один и тот же механизм (например, отслеживание онлайн-статуса пользователя или работа с локальным хранилищем) в нескольких разных компонентах, вы выносите эту логику в кастомный хук.

**Пример из вашего проекта:**
В папке `apps/frontend/src/shared/lib/hooks` находятся кастомные хуки:
- `useLocalStorage`: Инкапсулирует логику чтения и записи данных в `localStorage`, чтобы не писать `getItem` и `setItem` в каждом компоненте.
- `useWebSocket`: Управляет соединением с сервером, подписками и обработкой входящих сообщений, предоставляя компонентам простой интерфейс для работы с WebSocket.

**Итог:**
- **Встроенный хук** $\rightarrow$ инструмент от React для базовых функций.
- **Кастомный хук** $\rightarrow$ ваша обертка над встроенными хуками для повторного использования конкретной бизнес-логики.

### 5. Что такое провайдеры и роутеры? В чём отличие? Где приминяется и для чего?

**Провайдеры (Providers)** и **роутеры (Routers)** — это разные механизмы в React-приложении, хотя оба обычно подключаются на верхнем уровне приложения.

#### Что такое провайдер
**Провайдер** — это компонент-обертка, который передает общие данные, настройки или сервисы во всё дерево компонентов ниже себя.

Проще говоря: провайдер «раздаёт» контекст всему приложению или его части.

#### Для чего нужны провайдеры
- Чтобы не передавать данные вручную через props на каждом уровне.
- Чтобы подключить глобальные механизмы: авторизацию, тему, кэш запросов, сессию, локализацию.
- Чтобы централизовать общую логику и состояние.

#### Примеры из вашего проекта
В `apps/frontend/src/app/providers` уже есть:
- **QueryProvider**: подключает TanStack Query, чтобы компоненты могли работать с серверными запросами, кэшем и мутациями.
- **SessionProvider**: хранит и раздает данные о текущей пользовательской сессии.

То есть провайдер отвечает не за экран, а за **общий контекст и сервисы**, которыми пользуются экраны и компоненты.

#### Что такое роутер
**Роутер** — это механизм, который определяет, **какую страницу или экран показать в зависимости от URL**.

Например:
- `/login` -> экран входа
- `/register` -> экран регистрации
- `/room/123` -> страница комнаты

#### Для чего нужен роутер
- Для навигации между страницами без полной перезагрузки браузера.
- Для связи URL с конкретными страницами приложения.
- Для работы с параметрами маршрута, loader/action-логикой и защитой маршрутов.

#### Примеры из вашего проекта
В `apps/frontend/src/app/router` находятся:
- `routes.tsx`: описание маршрутов приложения
- `loaders/`: логика, которая выполняется до открытия страницы
- `actions/`: обработка действий, связанных с маршрутом, например отправка формы

То есть роутер отвечает не за общие данные, а за **навигацию и выбор страницы**.

#### В чём главное отличие
- **Провайдер** -> дает данным и сервисам доступ сверху вниз.
- **Роутер** -> решает, какой экран показать по адресу URL.

#### Как они работают вместе
Обычно приложение строится так:
1. Сначала на верхнем уровне подключаются провайдеры.
2. Затем внутри них подключается роутер.
3. Уже роутер показывает нужную страницу, а эта страница получает доступ к данным провайдеров.

Условная схема:

`QueryProvider -> SessionProvider -> Router -> Page`

#### Простой пример на практике
Если пользователь открывает `/dashboard`:
- роутер понимает, что нужно отрисовать `DashboardPage`;
- провайдер сессии может проверить, авторизован ли пользователь;
- провайдер запросов дает странице доступ к API и кэшу данных.

#### Итог
- **Провайдеры** нужны для передачи общего состояния, сервисов и конфигурации.
- **Роутеры** нужны для навигации между страницами и связи URL с экраном.
- **Провайдер не выбирает страницу**, а **роутер не хранит глобальное состояние приложения**.

### 6. Как полностью функционирует в frontend папка app? Для чего? Зачем? Почему?

Папка `app/` во frontend — это **ядро и точка сборки всего приложения**. Она не хранит конкретную бизнес-логику вроде голосования, комнат или задач. Ее задача — **инициализировать приложение, подключить глобальные зависимости, настроить маршруты, общие layout-оболочки и базовые стили**.

Проще говоря: если `features`, `entities`, `widgets` и `pages` — это содержимое приложения, то `app/` — это **каркас, в котором это содержимое начинает жить**.

#### Для чего нужна папка app
- Чтобы собрать приложение в единую систему.
- Чтобы вынести все глобальные настройки в одно место.
- Чтобы не размазывать роутинг, провайдеры, layout и стили по разным слоям.
- Чтобы верхний уровень приложения был предсказуемым и управляемым.

#### Почему app находится на верхнем уровне FSD
В FSD слой `app` — это самый верхний слой архитектуры. Он имеет право использовать нижележащие слои (`pages`, `widgets`, `features`, `entities`, `shared`), потому что именно он их связывает между собой.

Нижние слои не должны управлять приложением целиком. Они решают локальные задачи. А `app/` отвечает за **глобальную конфигурацию приложения**.

#### Как это работает в вашем проекте по шагам

1. **`main.tsx` запускает React-приложение**
   - Здесь создается корневой React-узел через `createRoot(...)`.
   - Затем рендерится компонент `<App />`.
   - Это самая первая точка входа frontend.

2. **`App.tsx` собирает верхний уровень**
   - Подключает глобальные стили из `app/styles/index.css`.
   - Оборачивает приложение в `AppProviders`.
   - Подключает `RouterProvider` с готовым объектом `router`.

   Условная схема:
   `main.tsx -> App.tsx -> AppProviders -> RouterProvider -> routes/pages`

3. **`app/providers/` подключает глобальные сервисы**
   - `QueryProvider.tsx` поднимает `QueryClientProvider` для TanStack Query.
   - Там же подключается `SessionProvider`, который управляет сессией пользователя.
   - `SessionProvider.tsx` хранит текущего пользователя, флаг загрузки, login/register/logout и синхронизирует токен через `SessionManager`.

   Это нужно, чтобы любой компонент ниже по дереву мог получить доступ к сессии или кэшированным серверным данным.

4. **`app/router/` управляет навигацией**
   - `routes.tsx` описывает все страницы: `/`, `/login`, `/register`, `/dashboard`, `/room/:roomId` и так далее.
   - `loaders/` выполняют проверки до открытия страницы.
   - `actions/` обрабатывают действия маршрутов, например отправку формы входа или регистрации.

   Например:
   - `authLoader` не пускает неавторизованного пользователя в `/dashboard`.
   - `publicOnlyLoader` не пускает уже авторизованного пользователя обратно на `/login`.

5. **`app/layouts/` задает общую оболочку страниц**
   - `RootLayout` добавляет общий `Header`, `Footer` и `Outlet` для вложенных страниц.
   - `AuthLayout` задает единый внешний вид для страниц входа и регистрации.

   Это нужно, чтобы страницы не дублировали один и тот же каркас вручную.

6. **`app/styles/` задает глобальный визуальный фундамент**
   - В `styles/index.css` подключается Tailwind.
   - Там задаются CSS-переменные темы, базовые цвета, фон, радиусы, dark-mode и общие анимации.

   Это слой не про конкретную кнопку, а про **общий визуальный язык всего приложения**.

#### Что именно хранить в app, а что нет

**В `app/` стоит хранить:**
- глобальные провайдеры;
- маршрутизацию;
- layout верхнего уровня;
- конфигурацию приложения;
- глобальные стили;
- bootstrap/инициализацию.

**В `app/` не стоит хранить:**
- бизнес-фичи;
- API конкретной сущности;
- UI конкретной страницы;
- локальную логику отдельных компонентов.

То есть:
- авторизация как глобальная сессия -> может быть в `app/providers`;
- форма логина как пользовательский сценарий -> уже `features/auth`;
- сама страница логина -> `pages/LoginPage`.

#### Зачем это архитектурно полезно
- Упрощает понимание входной точки приложения.
- Делает архитектуру чище: глобальное отдельно, бизнес-логика отдельно.
- Упрощает масштабирование: проще менять роутинг, сессию, тему, layout.
- Снижает хаос, когда приложение растет.

#### Итог
Папка `app/` во frontend нужна для того, чтобы **запустить, связать и настроить все приложение целиком**.

Она отвечает за:
- старт приложения;
- глобальные провайдеры;
- маршруты;
- layout-каркас;
- базовые стили и конфигурацию.

Если сказать совсем коротко:
- `pages/features/entities/widgets` отвечают за **содержимое**;
- `app/` отвечает за **сборку и управление приложением как системой**.

### 7. Расскажи подробно как работает каждый элемент в папке entities. Мы выносим отдельно index.ts чтобы было удобно импортировать из разных частей программы?

Да, **`index.ts` в `entities` выносится в первую очередь для удобного, чистого и контролируемого импорта**. Но это не единственная причина. Он еще формирует **публичный API сущности**: то есть явно показывает, что другим слоям разрешено использовать.

#### Что такое `entities/` в FSD
Слой `entities` хранит **бизнес-сущности приложения**. Это ответ на вопрос: **о чем приложение вообще говорит?**

В вашем проекте основные сущности:
- `user` — пользователь;
- `room` — комната для planning poker.

То есть здесь лежит не «любая логика», а код, который описывает **структуру данных сущности**, её базовые операции и иногда её базовое представление в UI.

---

#### 1. `entities/user/`

Эта папка отвечает за всё базовое, что связано с пользователем.

##### `entities/user/model/types.ts`
Здесь описаны **типы пользователя и аутентификации**:
- `IUser`
- `ILoginCredentails`
- `IRegisterCredentials`
- `ILoginResponse`
- `IRegisterResponse`
- `IGuestLoginResponse`

Зачем это нужно:
- чтобы фронтенд строго понимал, как выглядит пользователь;
- чтобы запросы логина/регистрации были типизированы;
- чтобы ответ от backend был предсказуемым.

То есть `model/types.ts` — это **контракт данных сущности**.

##### `entities/user/api/authApi.ts`
Здесь лежат **базовые API-операции**, связанные с пользователем:
- `login`
- `register`
- `getUser`
- `loginAsGuest`

Этот файл отвечает за запросы к backend через общий `api` клиент из `shared/api`.

Зачем это выносится сюда:
- чтобы логика HTTP-запросов не размазывалась по страницам и компонентам;
- чтобы любой слой выше мог использовать готовые функции работы с пользователем;
- чтобы вся работа с сущностью `user` была собрана рядом.

##### `entities/user/index.ts`
Этот файл делает **публичную точку входа** для сущности `user`.

Он реэкспортирует:
- API из `api/authApi`
- типы из `model/types`

Поэтому в других местах можно писать:
```ts
import { login, getUser, type User } from '@/entities/user';
```

вместо длинных импортов вроде:
```ts
import { login } from '@/entities/user/api/authApi';
import type { User } from '@/entities/user/model/types';
```

Это делает код:
- короче;
- чище;
- стабильнее при рефакторинге.

---

#### 2. `entities/room/`

Эта папка отвечает за сущность комнаты и всё базовое вокруг неё.

##### `entities/room/model/types.ts`
Здесь описаны **типы комнаты**:
- `RoomListItem`
- `RoomHistoryItem`
- `RoomSnapshot`
- `RoomDetails`
- а также типы `RoomState` и `RoomStatus` из `@poker/shared`

Это нужно, чтобы frontend знал:
- как выглядит комната в списке;
- как выглядит полное состояние комнаты;
- как выглядит история голосований;
- какие статусы бывают у комнаты.

`RoomSnapshot` здесь особенно важен: это большой снимок текущего состояния комнаты, который backend возвращает фронтенду.

##### `entities/room/api/roomApi.ts`
Это основной набор **операций над комнатой**:
- получение списка комнат;
- получение комнаты и snapshot;
- создание комнаты;
- вход по коду или приглашению;
- удаление комнаты;
- работа с задачами;
- запуск раунда;
- отправка голоса;
- reveal/finalize/reset.

То есть это **сервисный слой сущности room** на frontend.

Он нужен, чтобы все базовые действия над комнатой были централизованы и не дублировались в разных местах интерфейса.

##### `entities/room/model/selectors.ts`
Здесь лежат **селекторы и вспомогательная бизнес-логика**, которая вычисляет производные данные из сущности.

Например:
- `statusLabels` — человекочитаемые подписи статусов;
- `getRoomStatusColor` — какой цвет показывать для статуса;
- `isRoomActive` — активна ли комната.

Почему это не кладут прямо в компонент:
- потому что это логика, относящаяся к сущности, а не к конкретному экрану;
- потому что её можно переиспользовать в разных местах.

##### `entities/room/lib/roomSnapshotMappers.ts`
Этот файл содержит **мапперы**: он преобразует backend-данные комнаты в более удобный frontend-вид.

Например:
- `mapSnapshotTasks()` — превращает snapshot в массив задач;
- `mapSnapshotPlayers()` — превращает snapshot в игроков для UI;
- `resolveRoomOwnerName()` — вычисляет имя владельца;
- `getRoomVotingView()` — собирает удобную модель данных для экрана голосования.

Зачем это нужно:
- backend-структура часто слишком «сырая» для UI;
- компонентам лучше отдавать уже подготовленные данные;
- так уменьшается сложность самих страниц и виджетов.

##### `entities/room/ui/RoomCard/ui/RoomCard.tsx`
Это **базовый UI-компонент сущности room** — карточка комнаты.

Он показывает:
- slug комнаты;
- название;
- количество участников;
- последнюю активную задачу;
- действия входа, просмотра результатов и удаления.

Это находится в `entities`, потому что карточка напрямую представляет сущность `room`.

##### `entities/room/ui/RoomCard/index.ts`
Это локальный `index.ts` для короткого экспорта компонента:
```ts
export { RoomCard } from './ui/RoomCard';
```

##### `entities/room/index.ts`
Это главный публичный API сущности `room`.

Он реэкспортирует:
- `RoomCard`
- типы комнаты
- `roomApi`
- селекторы

Из-за этого другие части приложения могут импортировать сущность компактно:
```ts
import { RoomCard, roomApi, type RoomListItem } from '@/entities/room';
```

---

#### Зачем вообще нужны `index.ts`

Да, главная причина — **удобство импортов**, но архитектурно важнее другое: `index.ts` задает **границу модуля**.

Он решает сразу несколько задач:

1. **Упрощает импорт**
   - Не нужно помнить точные внутренние пути.

2. **Скрывает внутреннюю структуру**
   - Сегодня тип лежит в `model/types.ts`, завтра вы перенесете его в другой файл, а импорты по проекту не сломаются, если внешний `index.ts` останется прежним.

3. **Формирует публичный API**
   - Через `index.ts` вы явно решаете, что сущность отдает наружу.
   - Всё остальное считается внутренней реализацией.

4. **Упрощает рефакторинг**
   - Можно менять внутренние файлы без массового переписывания импортов.

5. **Поддерживает дисциплину FSD**
   - Другие слои должны пользоваться сущностью через ее публичный вход, а не лазить по внутренним файлам без необходимости.

---

#### Как это выглядит на практике

**Хорошо:**
```ts
import { getUser, type User } from '@/entities/user';
import { roomApi, RoomCard } from '@/entities/room';
```

**Менее желательно:**
```ts
import { getUser } from '@/entities/user/api/authApi';
import { RoomCard } from '@/entities/room/ui/RoomCard/ui/RoomCard';
```

Во втором случае внешний код слишком сильно зависит от внутреннего устройства папки.

---

#### Итог
Папка `entities` хранит **сущности предметной области**: их типы, базовые API-операции, вспомогательные вычисления и иногда базовый UI.

В вашем проекте:
- `entities/user` описывает пользователя и аутентификацию;
- `entities/room` описывает комнату, её состояние, операции и базовое отображение.

А `index.ts` нужен не только для удобства, но и для того, чтобы:
- сделать импорты короче;
- скрыть внутреннюю структуру;
- задать публичный API сущности;
- облегчить рефакторинг и поддержку архитектуры.

### 8. При разработке с проекта на стооне frontend, с чего стоит начинать разработку? С app, c entities? Или с чего вообще начинается разработка?

Если говорить правильно, то разработка на frontend **обычно начинается не с `entities` и не просто с папки `app` как таковой**, а с **понимания пользовательского сценария и точки входа приложения**.

Проще говоря:
- сначала нужно понять, **что именно должен уметь пользователь сделать**;
- потом определить, **какой экран, маршрут и UI для этого нужны**;
- и уже после этого раскладывать код по слоям FSD: `app`, `pages`, `widgets`, `features`, `entities`, `shared`.

То есть начинать стоит **не с папки**, а **с задачи и сценария**.

#### С чего начинается разработка frontend на практике

Обычно последовательность такая:

1. **Определяется пользовательский сценарий**
   - Что пользователь открывает?
   - Что он должен увидеть?
   - Что он должен нажать?
   - Какой результат ожидается?

   Например:
   - пользователь заходит на страницу комнаты;
   - видит список участников и задач;
   - выбирает карту;
   - отправляет голос;
   - видит итог после reveal.

2. **Определяется экран или маршрут**
   - Нужна ли новая страница?
   - Это отдельный URL или часть существующего экрана?
   - Нужен ли loader, guard, layout?

   На этом этапе уже появляется связь с `app/` и `pages/`.

3. **Собирается верхнеуровневый каркас**
   - если нужен новый маршрут -> меняется `app/router`;
   - если нужен новый layout или провайдер -> меняется `app/`;
   - если нужен новый экран -> создается `pages/...`.

   То есть `app/` чаще всего участвует в начале разработки как **точка интеграции**.

4. **Выделяются пользовательские действия**
   - логин;
   - создание комнаты;
   - вход в комнату;
   - отправка голоса;
   - завершение раунда.

   Это уже слой `features/`, потому что здесь описывается **что пользователь может сделать**.

5. **Выделяются сущности предметной области**
   - `user`;
   - `room`;
   - `task`;
   - `vote`.

   Это слой `entities/`.

   Но важно понимать: `entities` обычно удобно выделять **после того, как уже понятен сценарий и данные**, а не в полном отрыве от интерфейса.

6. **Выносятся общие части в shared**
   - кнопки;
   - модалки;
   - инпуты;
   - api client;
   - форматтеры;
   - общие хуки.

   То есть `shared/` появляется по мере того, как становится понятно, что код реально переиспользуется.

#### Нужно ли начинать именно с app

**Да, если речь о старте проекта в целом**, то разработка начинается с `app/`, потому что без него приложение вообще не запустится.

Именно в `app/` обычно сначала настраивают:
- `main.tsx`;
- `App.tsx`;
- роутинг;
- провайдеры;
- глобальные стили;
- layout верхнего уровня.

Без этого у вас не будет основы, в которую потом встраиваются страницы, фичи и сущности.

Но если проект **уже создан и инфраструктура готова**, то новую задачу вы начинаете не с переписывания `app/`, а с вопроса:

**какой пользовательский сценарий мы сейчас реализуем?**

#### Нужно ли начинать именно с entities

Обычно **нет, не стоит начинать с `entities` в первую очередь**.

Причина простая: если начать слишком рано строить `entities`, можно:
- создать лишние абстракции;
- придумать сущности, которые реально не нужны;
- вынести код слишком рано и усложнить структуру.

`entities` лучше выделять тогда, когда уже ясно:
- какие данные приходят с backend;
- какие поля реально используются;
- какие действия и представления нужны для этой сущности.

То есть `entities` — это **не стартовая точка любой задачи**, а **слой доменной модели**, который появляется осмысленно.

#### Правильный практический подход

Если упростить, то разработка frontend обычно идет сверху вниз:

1. `app/` — чтобы приложение запускалось и было куда встроить экран.
2. `pages/` — чтобы появился конкретный экран.
3. `widgets/` и `features/` — чтобы собрать экран и действия пользователя.
4. `entities/` — чтобы оформить доменные данные и базовую логику.
5. `shared/` — чтобы вынести переиспользуемые вещи.

Но в реальной работе это не всегда строго линейно.

Чаще всего процесс выглядит так:
- создали или открыли маршрут;
- собрали страницу;
- поняли, какие фичи нужны;
- поняли, какие сущности и типы нужны;
- вынесли общее в `shared`.

То есть разработка идет **от пользовательского сценария к архитектурной декомпозиции**, а не от абстрактной папки к папке.

#### Пример на вашем проекте

Допустим, нужно реализовать функционал «пользователь входит в комнату и голосует».

Тогда логика старта будет такой:

1. В `app/router` проверяем, есть ли маршрут комнаты.
2. В `pages/RoomPage` собираем экран комнаты.
3. В `features/voting` выносим действие голосования.
4. В `entities/room` используем типы комнаты, snapshot, API и селекторы.
5. В `shared/ui` используем или создаем общие кнопки, карточки, модалки.

То есть:
- **маршрут и сборка экрана** идут сверху;
- **сущности и общие элементы** подключаются ниже по мере необходимости.

#### Итог

Разработка frontend начинается:
- **не с `entities`**;
- **не только с `app`**;
- а с **понимания пользовательского сценария, экрана и точки входа в приложение**.

Если проект создается с нуля:
- сначала обычно поднимают `app/` как инфраструктурный каркас.

Если реализуется новая фича в уже существующем проекте:
- сначала определяют сценарий;
- потом страницу и маршрут;
- затем раскладывают логику по `features`, `entities` и `shared`.

Если сказать совсем коротко:
- `app/` — это старт инфраструктуры приложения;
- `entities/` — это слой доменной модели;
- реальная разработка начинается с **задачи пользователя и экрана**, а не с выбора папки «с чего начать».
